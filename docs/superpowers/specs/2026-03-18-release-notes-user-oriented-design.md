# Release Notes 用户导向统一化设计

## 背景

当前最近几版 GitHub Release Notes 存在明显风格漂移：

1. `v0.6.0`、`v0.6.1` 偏长，带有较强的设计说明和研发过程口吻。
2. `v0.7.0`、`v0.7.1` 又偏短，主要停留在“做了什么”，没有充分解释“为什么值得升级”。
3. 各版本结构不一致，用户难以快速建立阅读预期。
4. 实验性功能的说明边界不清晰：有时完全不提，有时又写成研发验证过程，影响用户理解。

用户明确要求后续 Release Notes 采用“用户导向”写法，但允许在确有必要时说明实验性功能的真实目的，包括“用于验证某条交互路径/工作流”，前提是这些表述服务于用户预期管理，而不是复盘研发过程。

## 目标

本次设计目标：

1. 统一最近几版 Release Notes 的结构和语气。
2. 让用户在几秒内判断“这版和我有没有关系”。
3. 在保留实验性信息的同时，避免写成内部设计说明。
4. 为后续版本沉淀一套可复用模板，减少每次发版时的随意发挥。

## 非目标

1. 不改动插件代码、测试或发布工作流。
2. 不追求把所有实现细节都搬进 Release Notes。
3. 不把 changelog 和 release notes 混为一体；Release Notes 仍以用户感知为中心。

## 执行对象与交付物

本次执行对象是 GitHub Releases 页面中以下 tag 的正文：

- `v0.7.1`
- `v0.7.0`
- `v0.6.1`
- `v0.6.0`
- `v0.5.0`

事实来源按以下优先级核对：

1. 当前 GitHub release 正文
2. 对应 tag 的提交与相关提交信息
3. 仓库内已有设计文档、README 或测试中能直接佐证的行为描述

本次交付物包括：

1. 更新后的五个 GitHub Release 正文
2. 一份后续可复用的 Release Notes 模板

## 目标读者

主要面向两类读者：

1. 已安装插件、正在考虑是否升级的用户。
2. 新看到 release 页面、想快速了解版本价值的潜在用户。

这意味着 Release Notes 需要优先回答：

- 这版解决了什么痛点？
- 谁应该关心这次更新？
- 升级后我会实际感受到什么变化？
- 有没有实验性边界或兼容性注意事项？

## 推荐写法

每个版本统一采用以下结构，按需裁剪：

### 1. 一句话价值总结

开头先用一句话说明该版本的直接价值，例如：

- 修复了什么高频问题
- 降低了什么使用成本
- 增加了什么新的可用入口

要求：

- 面向用户收益，不面向实现动作
- 1 句话即可，不展开技术细节

### 2. 适合谁升级

用 1-2 条说明升级相关人群，帮助用户快速筛选，例如：

- 如果你在官方 OpenCode release 里偶尔看不到 slash 命令
- 如果你希望在对话内直接查看 Copilot 配额
- 如果你正在使用实验性 guided loop safety 能力

这一段每版必须出现，至少 1 条，最多 2 条。

### 3. 你会看到的变化

列 3-5 条用户可感知变化：

- 输入某个命令后现在会发生什么
- 哪类报错或异常体验被修复
- 哪个操作路径变得更稳定、更少打断或更直接

写法规则：

- 多写“现在会怎样”，少写“新增/修改了什么函数”
- 不写内部模块名、hook 名、测试实现细节
- 保持可扫读，每条只表达一个结果

这一段每版必须出现，至少 2 条，最多 5 条。

### 4. 实验性功能

仅当该版本确实包含实验性能力时出现。

允许写的信息：

- 该能力目前是实验性的
- 当前主要面向哪个使用场景（如 TUI）
- 它的作用是否主要用于验证某条交互路径或工作流
- 开关位置、启用方式、稳定性边界

必须出现的条件：

- 当该版本包含实验性功能、实验性开关、实验性入口或稳定性边界时，必须写。

不应出现的条件：

- 当该版本只是普通修复，没有实验性能力时，不写这一段。

不建议写的信息：

- “本版额外目标是验证……”这类项目管理表述
- 过多解释研发决策过程

推荐句式：

- “这是一个实验性入口，当前主要用于验证……，因此……”
- “该能力目前主要面向……，如果你依赖……，可以先开启体验。”

禁用句式：

- “本版额外目标是验证……”
- “为了评估/论证/证明我们采用的方案……”
- “这次发布主要是给后续设计积累经验……”

合格示例方向：

- “这是一个实验性入口，当前主要用于验证 slash 交互路径，因此优先保证 TUI 可用，其他端形态后续可能继续调整。”

### 5. 兼容性 / 注意事项

只保留影响升级决策的内容，例如：

- 是否默认开启
- 是否提供关闭开关
- 哪些端形态不保证完全一致
- 是否需要特定配置

无重要事项时可省略，不必为了凑结构硬写。

## 语气与长度约束

统一写作约束如下：

1. 主体以 5-10 行可扫读内容为宜。
2. 语气直接、克制，不写研发复盘或过程感想。
3. 可以说明实验目的，但必须服务于用户预期管理。
4. 避免抽象空话，例如“增强稳定性”“优化体验”，除非后面紧跟具体表现。
5. 尽量使用“当你……时，现在会……”这种用户视角句式。

每版最低要求：

1. 必须回答“这版为什么值得升级”。
2. 必须回答“谁会关心这次更新”。
3. 必须回答“升级后用户会直接看到什么变化”。

## 最近几版的改写策略

本次需要统一改写以下版本：

- `v0.7.1`
- `v0.7.0`
- `v0.6.1`
- `v0.6.0`
- `v0.5.0`

每版改写前都要先做事实校验：

1. 对照当前 release 正文，确认原先承诺过的内容。
2. 对照对应版本的提交/标签，确认没有写入并不存在的效果。
3. 若某条描述无法从 release、提交、文档或测试中得到佐证，则不写。

各版本侧重点：

### `v0.7.1`

重点讲清楚 slash 命令偶发消失问题已修复，以及谁会明显受益。

### `v0.7.0`

重点讲清楚 `/copilot-inject`、`wait`、交互策略收敛对实际使用的意义；实验性能力需要明确标识。

### `v0.6.1`

重点讲清楚 `/copilot-status` toast 失败问题的用户影响和修复收益；避免写得像技术事故分析。

### `v0.6.0`

重点讲清楚实验性 `/copilot-status` 的实际使用价值，同时保留它主要用于验证 slash 交互路径的说明，但要从用户预期角度表述。

### `v0.5.0`

重点讲清楚 notify/question 分流、derived session 注入修复对使用中“少打断、更稳定”的帮助。

## 后续复用模板

后续新版本默认从以下模板起草：

```md
一句话说明：这版对用户最直接的价值。

## 适合谁升级
- 如果你……
- 如果你……

## 你会看到的变化
- …
- …
- …

## 实验性功能
- 这是实验性能力，目前……
- 它当前主要用于验证……，因此……

## 注意事项
- …
```

说明：

- `实验性功能` 和 `注意事项` 两段都允许省略。
- 如果一个版本主要是修复，正文可以缩短，但仍需先讲用户收益再讲变化。
- 无论如何，`适合谁升级` 与 `你会看到的变化` 两段不省略。

## 执行建议

1. 先按统一结构重写最近五个 GitHub Release 的正文。
2. 重写后复查是否满足以下检查项：
   - 第一屏能看出“为什么升级”
   - 实验性边界是否清楚
   - 是否避免了内部实现口吻
   - 各版结构是否基本一致
3. 后续发版时直接复用模板，必要时在发版命令或脚本中内置草稿骨架。

## 验收标准

完成后应满足：

1. `v0.7.1` 到 `v0.5.0` 的每版正文都包含：
   - 1 句价值总结
   - 1-2 条“适合谁升级”
   - 2-5 条“你会看到的变化”
2. 仅在确有实验性能力时出现 `实验性功能` 段落。
3. 实验性段落允许说明“用于验证某条交互路径/工作流”，但不得出现项目管理或研发复盘口吻。
4. 每版内容都已与对应 release、提交或现有文档事实核对，不包含无法佐证的说法。
5. 用户无需阅读代码或提交记录，就能理解该版本价值与升级相关性。
