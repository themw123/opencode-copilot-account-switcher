# v0.10.9-v0.11.0 Release Note 重写设计

## 背景

当前仓库自 `v0.10.9` 起的 release note 风格开始退化：

- 文案越来越接近机械生成的 `## Summary` 摘要；
- 用户视角与升级理由变弱；
- `v0.11.0` 甚至出现了字面量 `\n`，直接影响阅读体验。

与之相比，`v0.10.8` 的写法更像真实发布说明：先告诉用户“这版解决了什么问题、谁值得升级”，再列出用户可感知的变化。

本次工作的目标不是改代码，而是把一段已经退化的对外发布文案重新拉回到更自然、可读、面向用户的风格。

## 目标

1. 重写 `v0.10.9`、`v0.10.10`、`v0.10.11`、`v0.10.12`、`v0.11.0` 的 GitHub Release body。
2. 统一恢复到接近 `v0.10.8` 的写法：短导语 + 用户视角变化列表。
3. 让每个版本都能清楚体现自己的升级重点，而不是只剩模板化摘要。
4. 明确修复 `v0.11.0` 当前 body 中的字面量 `\n` 问题。

## 非目标

1. 不修改 git tag。
2. 不重新发布版本。
3. 不重写 commit 历史。
4. 不补做代码变更或文档变更来“配合” release note。

## 推荐写法

采用“短导语 + `## 你会看到的变化`”的统一结构。

### 结构

每个版本都采用：

1. 一段 1-3 句的短导语
   - 先说明这版主要解决了什么问题
   - 再点明谁会更想升级
2. 一个 `## 你会看到的变化` 小节
   - 3-4 条 bullet
   - 只写用户能感知到的结果

### 语气

- 句子短、直接、可扫读。
- 不写“修改了哪些文件”“新增了哪些测试”这类仓库内部动作。
- 必要技术名词保留原文，例如：
  - `AI_JSONParseError`
  - `/copilot-stop-tool`
  - `session.summarize(auto=true)`

### 与 `v0.10.8` 的关系

这次不是逐字模仿 `v0.10.8`，而是继承它的两个优点：

1. 先告诉用户“为什么值得升级”；
2. 再告诉用户“升级后你会看到什么变化”。

## 事实来源与约束

release note 的内容只能来自以下事实来源：

1. 相邻 tag 之间的提交信息；
2. 仓库里已有的 spec / plan 文档；
3. 测试名称与测试语义；
4. 现有 release body 中仍然可信的部分。

禁止：

- 编造仓库里没有的卖点；
- 夸大影响范围；
- 用“更强大”“更智能”之类空泛形容词替代具体变化。

## 版本级写作边界

### `v0.10.9`

- 保留其“hotfix / this binding 修复”的定位；
- 但改写成更自然的升级理由，而不是只剩机械摘要。

### `v0.10.10`

- 重点应放在账号路由语义收敛与 `unbound-fallback` 用户可见行为上；
- 不应写成纯内部分类术语清单。

### `v0.10.11`

- 重点应放在某类 Copilot 响应解析错误终于能自动重试；
- 要明确这不是“所有 parse error 都重试”。

### `v0.10.12`

- 重点应放在 `/copilot-status` 的展示体验提升；
- 要让用户能读懂“现在会看到怎样的分组配额视图”。

### `v0.11.0`

- 这是 feature release，重点应放在：
  - `/copilot-compact`
  - `/copilot-stop-tool`
  - `/copilot-status` 省略号修复
- 文案需要强调 stop-tool 是“更诚实的近似版”，不要暗示真正的 call-level tool cancel。
- 必须去掉当前 body 中的字面量 `\n`。

## 执行方式

1. 逐个读取 `v0.10.9` 到 `v0.11.0` 的现有 release body。
2. 逐个查看相邻 tag 之间的提交与文档线索。
3. 先写出每个版本的“升级理由一句话”。
4. 再补成完整的导语 + `## 你会看到的变化`。
5. 使用 GitHub Release 编辑能力原地更新这 5 个 release body。
6. 最后回读一次 release body，确认格式、换行和链接都正常。

## 验收标准

完成后应满足：

1. `v0.10.9` 到 `v0.11.0` 的 5 个 release body 都已更新。
2. 5 个版本都不再使用机械的 `## Summary` 风格。
3. `v0.11.0` 不再出现字面量 `\n`。
4. 每个版本都能让用户一眼看出“为什么升级”和“会感知到什么变化”。
5. 文案内容能被对应 tag 的提交、测试和文档事实支撑。
