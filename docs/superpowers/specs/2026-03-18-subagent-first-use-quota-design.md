# 子代理首用账号请求标识修正设计

## 背景

当前插件已经支持按 Copilot 模型把请求路由到不同账号：

- `src/plugin-hooks.ts` 中的 `fetchWithModelAccount()` 会读取请求体里的 `model`。
- 若 `resolveCopilotModelAccount()` 命中某个账号，就会用该账号的认证信息发出真实 Copilot 请求。

用户在实际使用中发现一个更关键的问题：

- 某个账号可能在当前 OpenCode 实例里从未真正发出过 Copilot 请求。
- 如果它第一次被实际命中恰好发生在子代理场景，当前请求会继承上游写入的 `x-initiator: agent`。
- 这会让该账号的第一次真实使用走到“agent 标识”路径，而不是用户希望的首次正常计费/配额触发路径。

因此，本次要解决的不是“额外补一次 quota 拉取”，而是修正首次真实请求本身的请求标识。

## 根因结论

根因已经收敛为两段行为叠加：

1. 上游 `opencode` 官方 Copilot 插件会在子代理 child session 中把请求标记为 `x-initiator: agent`。
   - 参考 `opencode/packages/opencode/src/plugin/copilot.ts:311`
   - 参考 `copilot-account-switcher/src/upstream/copilot-plugin.snapshot.ts:393`
2. 插件当前在 `fetchWithModelAccount()` 路径中只替换账号认证和请求目标域，不会修正这个 header。
   - 因此只要某账号第一次实际被命中发生在子代理里，该账号的首次真实请求就会带着 `agent` 标识发出。

这个问题与账号是否为当前 `active` 无关，关键在于：

- 该账号是否已经在当前 OpenCode 实例里真实发过请求。
- 以及它第一次真实请求是否来自带有 `x-initiator: agent` 的子代理路径。

## 目标

本次设计目标：

1. 按“账号在当前 OpenCode 实例中的第一次真实 Copilot 请求”来判断是否需要修正请求标识。
2. 若该账号的第一次真实请求带有 `x-initiator: agent`，则只对这一次请求移除该标识。
3. 同一账号后续请求恢复上游默认语义，不再持续干预。
4. 不修改 OpenCode core，只在插件侧实现。
5. 不增加额外 quota 补偿请求，不新增 toast，不改变现有菜单或 store 结构。

## 非目标

1. 不修改 OpenCode upstream/core 源码。
2. 不把“首次使用”状态持久化到 `store`。
3. 不新增菜单开关。
4. 不重写请求体内容。
5. 不长期禁用子代理的 `agent` 语义；只修正账号首用这一次。

## 用户已确认约束

1. 判断维度是“账号在当前实例里第一次被使用”，不是“非 active 账号第一次被使用”。
2. 即使某账号本身就是 active，只要它在当前实例里尚未真实发过请求，且第一次命中发生在子代理里，也应触发修正。
3. 不是每次都不发送 `agent` 标识，而是仅该账号第一次真实请求时不发送。
4. “当前实例”表示当前 OpenCode 进程生命周期内，不要求跨重启保留。

## 推荐方案

采用“按账号名做实例内一次性首用抑制”的方案。

实现思路：

- 在 `buildPluginHooks()` 闭包中维护一个进程内 `Set<string>`。
- 每次真实发请求前，按最终将被用于请求的账号名判断它是否已经在当前实例里使用过。
- 如果没有使用过，且当前 header 明确为 `x-initiator: agent`，则只对这一次请求移除该标识。
- 一旦该请求进入真实发送路径，就把该账号记为“已使用”。
- 之后该账号再被命中时，不再做任何特殊处理，完全恢复上游默认 header。

推荐原因：

- 与用户确认的“当前实例、按账号、一次性”语义完全一致。
- 只改插件自身的请求封装层，不需要改上游判定逻辑。
- 改动最小，风险最可控。

## 架构设计

### 1. 状态位置

在 `src/plugin-hooks.ts` 的 `buildPluginHooks()` 闭包中新增：

- `usedCopilotAccounts = new Set<string>()`

语义：

- key 为最终发请求时使用的账号名。
- 只存在于当前插件进程内存中。
- 不写入 `store`。

这样可以确保：

- 同一 OpenCode 实例里，同一账号只会触发一次首用修正。
- 进程重启后重新统计，符合“当前实例”定义。

### 2. 判定时机

判定逻辑放在请求最终选定账号之后执行，不能只局限在“模型映射命中”分支。

当前已有流程：

1. 读取请求体中的 `model`
2. `resolveCopilotModelAccount(store, modelID)`
3. 用目标账号构造认证信息
4. 调用官方 `config.fetch()` 发请求

新增流程：

1. 确定本次最终使用的账号名 `resolved.name`
2. 检查 `usedCopilotAccounts` 是否已包含该账号
3. 若未包含，则把这一次请求视为该账号在当前实例中的首次真实请求
4. 仅当当前 header 为 `x-initiator: agent` 时，对该次请求临时移除这一项
5. 然后继续沿原路径发出真实请求
6. 请求一旦进入发送路径，即把该账号加入 `usedCopilotAccounts`

这里的“首次”不区分来源：

- 可以来自模型映射命中账号
- 也可以来自 active 账号默认回退

只要最终真实发请求的是这个账号，就会消耗掉它在本实例里的“第一次”。

这意味着实现不能继续保留当前 `resolved.source !== "model"` 就直接早退到 `config.fetch(request, init)` 的结构。

推荐收口方式：

- 仍在 `fetchWithModelAccount()` 中完成账号解析与请求发送。
- 但在拿到 `resolved` 之后，无论 `resolved.source` 是 `model` 还是 `active`，都统一走“首用判定 -> 必要时修正 header -> 再发送请求”的路径。
- 只有在根本无法解析出任何账号时，才直接退回 `config.fetch(request, init)`。

### 3. Header 修正策略

仅对首次请求做最小修正：

- 仅当当前请求 headers 中存在 `x-initiator: agent` 时，移除这一项
- 若 `x-initiator` 为 `user` 或其他值，则保持原样
- 若该 header 本来不存在，则保持原样
- 其余 headers 保持不变
- 请求 body 不改
- URL 和认证替换逻辑保持不变

这样做的原因：

- 当前已确认问题集中在 `x-initiator: agent`
- 不需要引入更大范围的请求改写
- 可最大程度保持与上游行为兼容

### 4. 记账时机

推荐在“请求准备进入真实发送路径”时就把账号记入 `usedCopilotAccounts`。

也就是：

- 当插件已经解析出目标账号，并完成首次 header 修正后
- 在调用 `config.fetch(...)` 之前，把账号写入 Set

原因：

- 用户要的语义是“第一次被使用”，而不是“第一次成功返回”。
- 即使请求后续返回 4xx/5xx，它也已经构成该账号在当前实例中的一次真实使用尝试。
- 这样可以避免并发或重试导致同一账号反复被当成“首次”。

### 5. 与上游子代理语义的关系

这个设计不是禁用上游的子代理标识逻辑，而是做一次性首用修正：

- 对尚未使用过的账号，第一次真实请求不沿用 `agent` 标识
- 对已使用过的账号，后续继续保留上游对 child session / compaction 的默认处理

因此：

- 上游原生子代理语义仍然存在
- 只是插件在“账号首用”这个特殊窗口做了一次局部覆写

## 备选方案

### 方案 2：把首用状态写入 store

做法：

- 在 `store` 中增加按账号记录的首用状态
- 跨进程、跨重启保留

优点：

- 状态稳定，不受重启影响

缺点：

- 不符合用户确认的“当前实例”语义
- 增加持久化状态和迁移成本
- 把运行态一次性信息写进长期 store，不够干净

因此不推荐。

### 方案 3：所有模型映射账号请求都移除 `agent`

做法：

- 只要请求走模型映射账号，就始终移除 `x-initiator`

优点：

- 实现最简单

缺点：

- 语义过强，会持续偏离上游行为
- 与用户确认的“只有第一次不发送”不一致

因此不推荐。

## 错误处理与并发约束

1. 若请求没有 `x-initiator`：
   - 首次请求仍记为已使用
   - 不做额外改写

2. 若请求的 `x-initiator` 为 `user` 或其他非 `agent` 值：
   - 首次请求同样会消耗该账号的“第一次”
   - 但不会改写 header

3. 若请求已经不是子代理路径：
   - 首次请求同样会消耗该账号的“第一次”
   - 后续即使首次子代理请求到来，也不再触发抑制

4. 并发首用：
   - 若两个请求几乎同时命中同一账号，先进入发送路径的那个请求会先写入 Set
   - 后续并发请求将视为非首次，不再移除 `x-initiator`
   - 这是可接受的最小实现

5. 请求失败：
   - 不回滚 Set
   - 因为该账号已经发生过一次真实使用尝试

## 测试设计

主要新增 `test/plugin.test.js` 用例：

1. 某账号第一次请求来自子代理，且 header 原本是 `x-initiator: agent` 时，这一次会被移除。
2. 同一账号第二次再次走子代理时，`x-initiator: agent` 会恢复保留。
3. 不同账号分别维护各自的首用状态：
   - 账号 A 已经消耗过首次机会，不影响账号 B 第一次请求仍触发移除。
4. 首次请求若 `x-initiator: user`，则保持不变，但仍会消耗该账号的首次机会。
5. 首次请求若根本没有 `x-initiator`，则保持不变，但仍会消耗该账号的首次机会。
6. 首次请求即使不是模型映射，而是 active 账号默认路径，也会消耗该账号的首次机会。

不需要新增：

- UI 测试
- store 迁移测试
- quota/toast 测试

## 涉及文件

- `src/plugin-hooks.ts`
  - 新增进程内账号首用状态
  - 在 `fetchWithModelAccount()` 中实现一次性 `x-initiator` 移除
- `test/plugin.test.js`
  - 补充账号首用请求标识修正测试

## 预期结果

完成后，在同一个 OpenCode 实例中：

1. 某账号第一次真实发出 Copilot 请求时，不再继承子代理带来的 `x-initiator: agent`。
2. 这次修正只发生一次。
3. 同一账号后续请求恢复上游默认语义。
4. 不改 OpenCode core，不引入新的持久化状态，也不额外补 quota 请求。
