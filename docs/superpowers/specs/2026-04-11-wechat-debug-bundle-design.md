# WeChat 一键调试包导出设计

## 背景

当前仓库已经有较完整的 WeChat 本地状态与诊断体系，包括：

- `broker.json`
- `tokens/`
- `notifications/`
- `requests/`
- `dead-letter/`
- `instances/`
- `wechat-status-runtime.diagnostics.jsonl`
- `wechat-broker.diagnostics.jsonl`
- `wechat-bridge.diagnostics.jsonl`
- `broker-startup.diagnostics.log`

当用户反馈“微信通知没有效果”时，人工指导对方逐个找目录、复制文件、再区分哪些内容能发、哪些内容需要脱敏，成本很高，也容易漏掉关键信息。

因此需要一个插件内置的“一键导出微信调试包”能力，让用户在菜单里直接触发，并自动生成压缩包，降低排障门槛。

## 目标

1. 在插件菜单里提供一个明确的 `导出微信调试包` 入口。
2. 支持两种导出模式：`脱敏包` 与 `完整包`。
3. 自动收集 WeChat 状态目录、诊断文件、环境摘要与导出清单，并打包成 zip。
4. 对失败路径给出稳定中文提示，而不是留下半包或静默失败。
5. 让后续可能的外部脚本或别的入口也能复用同一套采集核心。

## 非目标

1. 本轮不做远端上传、分享链接或自动提交 issue。
2. 不做调试包内容预览 UI。
3. 不做脱敏规则的交互式逐项选择。
4. 不收集 `node_modules`、大型无关缓存目录或整个仓库源码快照。

## 方案选择

### 方案 A：插件内置导出器

做法：

- 在插件菜单里增加一组 WeChat 调试导出动作。
- 插件内部直接完成采集、脱敏、manifest 生成和 zip 打包。

优点：

- 用户路径最短，最符合这次“直接做在插件里”的要求。
- 采集口径统一，不依赖用户手工执行脚本。
- 后续若要补外部脚本，也可以复用同一采集核心。

缺点：

- 插件里需要承担 zip 打包和文件枚举逻辑。

### 方案 B：插件触发临时脚本

做法：

- 菜单动作只负责生成并执行一个临时脚本，导出逻辑主要留在脚本侧。

优点：

- 看起来实现拆分更清楚。

缺点：

- 对用户来说是“插件入口下再套一层脚本”，交付面不直观。
- 出问题时更难判断是插件逻辑还是脚本逻辑。

### 方案 C：最小目录打包

做法：

- 只把 `wechat/` 状态根目录原样压缩，不额外生成 manifest 或环境摘要。

优点：

- 实现最快。

缺点：

- 对排障帮助有限，尤其是无法快速分辨导出来源、模式与脱敏情况。

### 结论

采用方案 A：插件内置导出器。

## 总体架构

这条功能拆成两层：

1. **采集核心**
   - 一个插件内可复用的 collector 模块
   - 负责枚举文件、执行脱敏、生成 manifest、创建 zip
2. **插件入口**
   - 在现有插件菜单里增加 `导出微信调试包`
   - 先询问用户选择 `脱敏包` 或 `完整包`
   - 调用 collector，最终返回压缩包路径

这样当前交付面是插件菜单，但内部能力可复用，不会把实现绑死在某一个入口上。

## 导出内容

调试包固定收集四类信息：

### 1. 状态文件

来自 `wechatStateRoot()` 下的以下内容：

- `broker.json`
- `tokens/`
- `notifications/`
- `requests/`
- `dead-letter/`
- `instances/`

### 2. 诊断文件

- `wechat-status-runtime.diagnostics.jsonl`
- `wechat-broker.diagnostics.jsonl`
- `wechat-bridge.diagnostics.jsonl`
- `broker-startup.diagnostics.log`

### 3. 环境摘要

至少包括：

- 插件版本
- Node 版本
- 平台信息
- 当前工作目录
- 当前 git HEAD commit
- 当前导出模式（脱敏/完整）
- WeChat 状态根是否存在
- 各关键目录/文件是否存在

### 4. 导出清单

新增 `manifest.json`，记录：

- 导出时间
- 导出模式
- 每个收集到的文件的原始来源路径
- 是否做过脱敏
- 若文件不存在，是否被跳过

## 脱敏与完整模式

### 脱敏包

默认对以下字段做定向替换：

- `contextToken`
- 各类 token / bearer / cookie / credential 字段
- `wechatAccountId`
- `userId`
- 可能含敏感内容的完整消息正文或原始文本片段

替换方式应稳定且可识别，例如：

- `[REDACTED_CONTEXT_TOKEN]`
- `[REDACTED_ACCOUNT_ID]`
- `[REDACTED_USER_ID]`
- `[REDACTED_MESSAGE_TEXT]`

### 完整包

保留原始值，但仍然排除明显无关的大目录和文件，例如：

- `node_modules`
- 仓库整体源码快照
- 与 WeChat 调试无关的全局缓存目录

## 菜单交互

插件菜单只新增一个主入口：

- `导出微信调试包`

触发后先让用户二选一：

1. `脱敏包`
2. `完整包`

然后直接在本机生成 zip，成功后返回固定中文结果，例如：

```text
微信调试包已生成：C:\...\wechat-debug-bundle-2026-04-11T08-30-00.zip
```

失败时返回稳定中文错误，不保留半包，例如：

- `微信状态目录不存在，无法导出调试包`
- `没有可导出的微信诊断文件`
- `创建压缩包失败：<错误摘要>`

## 文件输出位置

建议默认输出到系统临时目录下的专用目录，例如：

```text
<temp>/opencode-copilot-account-switcher/wechat-debug-bundles/
```

压缩包名称带时间戳和模式，例如：

- `wechat-debug-bundle-sanitized-2026-04-11T08-30-00.zip`
- `wechat-debug-bundle-full-2026-04-11T08-30-00.zip`

## 错误处理

### 1. 缺目录/缺文件

- 单个文件缺失不应直接失败
- 但若整个 `wechatStateRoot()` 不存在，则应返回明确失败

### 2. 脱敏失败

- 任何文件的脱敏异常都应中止当前导出，并返回错误
- 不允许生成“部分脱敏、部分原样”的不确定包

### 3. 压缩失败

- zip 失败应删除未完成的输出文件
- 不给用户返回一个不可用路径

## 测试策略

至少覆盖：

1. collector 能正确枚举状态文件与诊断文件
2. `脱敏包` 会替换敏感字段
3. `完整包` 保留原始值
4. `manifest.json` 内容稳定且完整
5. 菜单入口能触发导出并返回 zip 路径
6. 缺失状态目录时返回稳定中文错误
7. zip 失败时不会留下半成品输出

## 成功判定

完成后应满足：

1. 用户可直接从插件菜单导出微信调试包
2. 默认同时支持 `脱敏包` 与 `完整包`
3. 调试包内包含状态文件、诊断文件、环境摘要和 manifest
4. 失败路径有明确中文提示，不留半包
5. 采集核心可被后续其他入口复用，但当前主入口仍是插件菜单
