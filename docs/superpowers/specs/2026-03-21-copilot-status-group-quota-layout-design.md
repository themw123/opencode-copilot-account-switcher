# /copilot-status 分组配额布局设计

## 背景

当前 `/copilot-status` 的成功 toast 仍然把 `store.active` 当作主视角：

- 第一行展示 `current active`
- 同时展示 `premium`、`chat`、`completions`
- 后两行只是把默认组和路由组中的账号名原样列出来

这和当前插件的实际使用方式不一致。

用户真正关心的是：

1. 默认组里有哪些账号，各自还剩多少 `premium requests`
2. 每个模型路由组里有哪些账号，各自还剩多少 `premium requests`

用户并不需要再次确认自己配置过哪些账号名，也不需要 `/copilot-status` 再强调 `active` 这个旧概念。

## 目标

1. 把 `/copilot-status` 成功 toast 改成“分组 premium 配额视图”。
2. 不再单独展示 active 账号，也不再展示 `chat` / `completions`。
3. 让分组内容在窄 toast 中保持整齐、可预测、不依赖自然换行运气。
4. 保持当前 quota 刷新入口不变，本轮只收敛展示语义和相关错误文案。

## 非目标

1. 不重做 quota 刷新机制。
2. 不在本轮移除 `store.active` 或修改 `auth.json` 同步策略。
3. 不扩展成真实使用轨迹、最近切换历史或账号统计面板。
4. 不增加除 `premium` 之外的新 quota 展示维度。

## 用户确认约束

### 1. 展示内容

- 成功 toast 只展示分组中的 `premium` 配额。
- 同一账号在多个分组中允许重复出现。
- 没有 quota 的账号显示 `n/a`。

### 2. 分组语义

- 默认组标题使用 `[default]`。
- 每个模型路由组标题直接使用 `[model-id]`。
- 组与组之间不插入空行。
- 如果没有任何模型路由组，显示单独一行 `[routes] (none)`。
- 默认组内账号顺序严格跟随 `store.activeAccountNames` 的现有顺序，不做额外排序。
- 模型路由组的分组顺序按 `modelAccountAssignments` 的 key 做字典序升序输出，保证测试和展示稳定。
- 每个模型路由组内的账号顺序严格跟随该模型对应数组里的现有顺序，不做额外排序。

### 3. 排版约束

- 账号区按 `50` 宽设计。
- 每行固定 `3` 列均分。
- 整体列宽稳定，不允许靠自然换行凑结果。
- 动态宽度只发生在单个账号块内部：quota 实际越宽，可用于用户名的空间就越窄；但不会影响同一行另外两列的宽度。
- 用户名超出可用宽度时，采用“中间省略”，例如 `verylo...name`。

## 推荐方案

采用“保留现有刷新入口 + 重写成功 toast 布局 + 去 active 化错误文案”的最小收敛方案。

### 1. 成功 toast 改成分组视图

成功消息不再包含：

- `current active`
- `chat`
- `completions`
- `updated at`

而是改成按组输出：

```text
[default]
alice 300/300   bob 20/300   carol n/a
[gpt-5.4]
bob 20/300      carol n/a    dave 2000/2000
[claude-3.7]
alice 300/300
```

上面的内容只是结构示意；真正实现时要满足下面的固定排版规则。

### 2. 50 宽三列排版规则

账号区采用固定宽度网格：

1. 每行总宽预算为 `50`
2. 每行固定放 `3` 个账号块
3. 三列宽度固定为 `16 + 1 + 16 + 1 + 16`，其中两个列间分隔符都为单个空格，总宽正好 `50`
4. 也就是说，每个账号块的外部宽度恒为 `16`
4. 每个账号块内部格式为：
   - `username`
   - `1` 个空格
   - `quota`
5. `quota` 宽度不做全局硬编码，而是按该账号实际显示值占用：
   - 常见：`300/300`
   - 少数：`2000/2000`
   - 缺失：`n/a`
6. 在单个账号块内部，quota 占用越宽，用户名可用宽度越窄；这种动态收缩只影响这个账号块本身
7. 账号块内部的可用用户名宽度按 `16 - quota.length - 1` 计算；这里的 `1` 是用户名和 quota 之间必须保留的那个空格
8. 如果可用用户名宽度大于 0，就输出 `username + 空格 + quota`
9. 如果可用用户名宽度小于等于 0，优先保留 quota：该账号块退化为只显示 quota，并在 `16` 宽账号块内右对齐；用户名本轮直接省略
10. 如果 quota 自身长度超过 `16`，仍然不允许撑破列宽；该账号块保留 quota 的右侧 `16` 个字符并右对齐，用户名省略
11. 当用户名宽度不足时，使用中间省略
12. 每一行在逻辑上始终按 3 个账号块排版；如果末行不足 3 个账号，就在右侧补齐空白账号块，保证这行的字符预算和列对齐规则与完整三列行一致

这样可以同时满足：

- 三列整体对齐
- 50 宽内可预测
- 稀有长 quota 值不会破坏整行结构
- 用户名仍保留前后缀识别性

### 3. 空组显示

为了避免用户看到“没有输出，不知道是空还是坏了”，空组显示规则定为：

- `store.activeAccountNames` 不存在、不是数组、为空数组，或归一化后没有任何有效账号时，都视为“没有默认组”
- `store.modelAccountAssignments` 不存在、为空对象，或所有模型分组在归一化后都没有任何有效账号时，都视为“没有任何模型路由组”

- 没有默认组时：

```text
[default]
(none)
```

- 没有任何模型路由组时：

```text
[routes] (none)
```

这里把无模型路由组视为“一个整体空态”，而不是再伪造不存在的模型分组。

### 4. 错误文案去 active 化

当前 `No active account available for Copilot status.` 这类文案会误导用户，以为 `/copilot-status` 的核心仍然是 active 账号。

本轮应改成更贴近新语义的表达，例如：

- 没有可用于刷新 Copilot 配额的默认账号
- 无法读取默认组账号，因此不能刷新 Copilot 配额

关键要求：

- 允许内部仍依赖现有刷新入口
- 但对用户可见的文案不要再把 active 当成主要概念

### 5. 刷新行为先保持不变

本轮不改变：

- `refreshActiveAccountQuota(...)` 的入口与返回结构
- 现有 store 写回流程

原因：

1. 用户这轮重点是展示语义错误，而不是 quota 刷新机制本身。
2. 如果同时重做刷新模型，会把任务扩大成“展示层 + 数据刷新层”双重改造。
3. 当前最小可交付是：继续刷新一次现有入口，再把 store 中各分组账号的 `premium` 快照按新布局渲染出来。

## 备选方案

### 方案 B：继续保留 active 行，只减少字段

例如：

```text
current active: alice premium 300/300
[default]
...
```

不推荐原因：

- 仍然在视觉上把 active 放在最重要位置
- 和用户想要的“按组看配额”相冲突

### 方案 C：继续单行拼接默认组与路由组

例如：

```text
[default] alice 300/300, bob 20/300
[gpt-5.4] bob 20/300, carol n/a
```

不推荐原因：

- 账号多时会依赖自然换行
- 很难在窄 toast 中稳定保持整齐

## 测试策略

本轮应至少覆盖以下行为：

1. 成功 toast 不再包含 `current active`、`chat`、`completions`
2. 成功 toast 会按 `[default]` 和 `[model-id]` 输出分组
3. 同一行按三列等宽布局输出多个账号块
4. quota 为 `2000/2000` 时仍能保持布局可控
5. 用户名超宽时会走中间省略
6. 没有默认组时显示 `[default]` + `(none)`
7. 没有任何模型路由组时显示 `[routes] (none)`
8. 旧的 active 缺失错误文案被替换成去 active 化表达

## 风险与边界

1. TUI toast 是否严格按等宽字体渲染，不完全由插件控制；本轮方案的“等宽”本质是字符预算稳定，而不是依赖真正的表格组件。
2. 如果未来 quota 文案又出现比 `2000/2000` 更长的极端值，账号块内部的用户名空间会进一步缩小，但整体布局仍应保持在 50 宽预算内。
3. 本轮仍复用 active 刷新入口，因此“展示不再强调 active”和“内部仍沿用该入口”之间会有一个过渡期；这是明确接受的折中。

## 结论

本次 `/copilot-status` 调整采用最小但语义正确的方案：

- 成功 toast 改成分组 premium 配额视图
- 默认组显示为 `[default]`，模型组显示为 `[model-id]`
- 账号区按 50 宽、三列均分、账号块内部动态压缩的规则排版
- 用户名超宽时用中间省略
- 空组显示显式 `(none)`
- 错误文案去 active 化
- 刷新机制暂不扩 scope，只修正展示与文案
